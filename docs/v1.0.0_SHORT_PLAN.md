## tree2guide — Roadmap & Execution Plan

### Guiding principle
Ship something real at each stage rather than redesigning everything up front. The single-file script already works — don't break that while restructuring.

---

### Phase 0 — Stabilize the current script (do this first, low risk)
Before any packaging work, lock in correctness and fix the one real bug flagged in review.

- **Fix `has_included_descendant()` performance** — currently re-walks subtrees repeatedly (worst-case O(n²) on deep/wide trees). Fold inclusion-checking into the single recursive `walk()` pass instead of a separate lookahead pass.
- **Add a small test suite** (`pytest`) covering the `GitignoreRule`/`ExcludeMatcher` logic — anchoring, `**`, dir-only, negation — using the test cases already proven out earlier in this conversation as a starting point.
- **Type hints pass** — `md_lines: list[str] = []` style cleanup throughout, per review.

This phase is pure hardening — no new features, no new files for the user. Output stays identical.

---

### Phase 1 — quality-of-life flags (still single-file, low risk)
Small, additive CLI flags that don't change the architecture:

| Flag | Purpose |
|---|---|
| `--stdout` | Print markdown to stdout instead of writing a file (enables `\| pbcopy`, `\| clip`, piping) |
| `--max-depth N` | Cap recursion depth — critical for huge monorepos |
| `--dirs-only` / `--files-only` | Filter the tree to one type |
| `--no-hidden` | Skip dotfiles/dotfolders without needing `.tree2ignore` entries for them |
| `--sort {alpha,dirs-first,files-first}` | Control ordering instead of hardcoded dirs-then-files |
| Symlink handling | Detect `is_symlink()`, render `name -> target` instead of silently recursing into it |

Each of these is independent — can be done incrementally and tested in isolation.

---

### Phase 2 — Package it properly (`pip install tree2guide`)

This is the first real structural change, and it's worth designing around a **Scanner → Tree Model → Renderer** separation from the start, rather than bolting formats onto one function later:

```
tree2guide/
├── src/
│   └── tree2guide/
│       ├── __init__.py
│       ├── cli.py            # argparse, main()
│       ├── ignore.py         # GitignoreRule, ExcludeMatcher (the Scanner's filter)
│       ├── scanner.py        # filesystem walk -> builds the Tree Model
│       ├── tree_model.py     # internal representation: {name, type, children}
│       └── renderers/
│           ├── markdown.py   # default renderer
│           ├── text.py
│           ├── json_renderer.py
│           ├── yaml_renderer.py
│           └── html.py
├── tests/
│   └── test_ignore.py
├── pyproject.toml
├── LICENSE
├── README.md
└── CHANGELOG.md
```

**Conceptually:**

```
Scanner (filesystem + ignore parser)
        ↓
   Tree Model (internal representation, built once)
        ↓
   Renderer (Markdown / Text / JSON / YAML / HTML)
```

The scanner builds the tree **once**; each renderer just transforms that same object into a different output format. This is what makes formats like `--format json` and `--format html` cheap to add later — they're new renderers, not new scanning logic.

- `pyproject.toml` defines a console entry point so `tree2guide .` works directly after install, no `python` prefix.
- Publish to PyPI once the module split + entry point are verified working locally (`pip install -e .`).
- Backward compatibility matters here — anyone who downloaded the old single `tree2guide.py` should still be able to use it, or get a clear migration note in the README.

---

### Phase 3 — Multi-format output

With the Scanner/Tree Model/Renderer split in place from Phase 2, this phase is mostly just **writing new renderers**, not new architecture:

```bash
tree2guide . --format markdown   # default
tree2guide . --format text
tree2guide . --format json
tree2guide . --format yaml
tree2guide . --format html
```

Markdown stays the **default output**, not the only output — important for positioning (see below). JSON output looks like:

```json
{
  "name": "src",
  "children": [
    { "name": "api", "type": "directory" }
  ]
}
```

This JSON form is also the natural backbone for the `--llm` mode in Phase 4, and for any future web-based interactive visualization.

---

### Phase 4 — The differentiator: `--llm` mode

```bash
tree2guide . --llm
```

Two parts:
1. **Structure** — same tree, but in a deliberately AI-consumption-friendly format. Easy to build since it reuses Phase 3's Tree Model — it's really just another renderer.
2. **Summary heuristics** — lightweight, dependency-free detection:
   - Framework/language signals from filenames (`pyproject.toml`, `package.json`, `requirements.txt`, `go.mod`, etc.)
   - Counting things (route files, model files, test files) via simple heuristics per detected stack
   - Flagging presence of `tests/`, `.env.example`, CI config, etc.

Keep this rule-based and transparent at first (no actual LLM call, no network dependency) — stays zero-dependency and fast. An optional "call an LLM API for a richer summary" mode could be a stretch goal, clearly opt-in and clearly costing the user an API key/credits.

---

### Phase 5 — Docs site + playground (parallel track, not blocking)

Can start as soon as Phase 2's package structure stabilizes, since the site's "Quick Start" and ".tree2ignore Guide" sections need the final CLI surface to write against.

- Static docs site (Home, Installation, Quick Start, `.tree2ignore` Guide, CLI Options, Examples, API Reference, Contributing) — markdown-driven, deployed wherever's easiest (GitHub Pages is the zero-cost default given the project is already on GitHub).
- Browser playground (paste a folder structure or upload a ZIP, get instant Markdown output) is the most expensive piece — needs either a backend or a WASM/Pyodide-based in-browser Python runtime. Treat as a stretch goal after the core docs site ships; it's the part most likely to balloon in scope.
- If Phase 3's JSON renderer exists, the playground/website could also render the tree **interactively** (collapsible folders, etc.) rather than just showing static Markdown — worth keeping in mind when designing the JSON shape.

---

### Positioning & Naming

**Don't market this as a "GitHub tool" — market it as a project documentation, visualization, and self-review tool.**

Markdown is much bigger than GitHub. It's used in GitLab, Bitbucket, Azure DevOps, Notion, Obsidian, VS Code, JetBrains IDEs, MkDocs, Docusaurus, Hugo, static site generators, AI prompts, and internal company docs. Markdown is the right **default**, not a ceiling — which is exactly why Phase 3's multi-format renderer architecture matters: it lets the tool grow into "Project Structure Generator" territory (JSON/YAML/HTML output, interactive web visualization) without ever needing Markdown to be the only thing it does.

**On the name `tree2guide`:** unlike `tree2md`, this name doesn't lock the project into one output format — "guide" describes the *purpose* (helping you understand and evaluate your own structure), not a file extension. That makes it more durable as the project grows into JSON/YAML/HTML renderers and the `--llm` mode, since none of those contradict what "guide" promises. It also captures the personal origin story directly: a guide for developers who don't have time to study every architecture principle before they start building, but still want to catch bad structure early.

---

### Suggested order of attack
1. **Phase 0** (fix the real bug, add tests) — do this now, it's small and protects everything after it
2. **Phase 1** (flags) — fast wins, no architecture risk
3. **Phase 2** (packaging + Scanner/Tree Model/Renderer split) — the real lift, unlocks PyPI *and* sets up every future format for free
4. **Phase 3 → 4** (multi-format renderers → `--llm`) — cheap once Phase 2's architecture exists
5. **Phase 5** (docs site → playground) — parallel track, start once Phase 2's CLI surface is stable
